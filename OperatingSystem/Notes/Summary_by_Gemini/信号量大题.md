这道题（2014年考研操作系统大题）是在经典的“生产者-消费者”模型基础上，增加了一个**消费者批量取件**的限制条件。

解决这道题的核心在于：既要保证基本的生产者-消费者同步逻辑，又要通过一个额外的**互斥信号量**来锁定“连续取10件”的行为，确保在此期间没有其他消费者能插队。

以下是该题的标准答案写法及解析：

### 1. 信号量定义与初值

- **`empty = 1000`**: 资源信号量，代表环形缓冲区中的空位数量。
- **`full = 0`**: 资源信号量，代表缓冲区中已存放的产品数量。
- **`mutex = 1`**: 互斥信号量，用于保证生产者和消费者互斥地访问缓冲区（即修改缓冲区指针）。
- **`mutex_consumer = 1`**: 互斥信号量，用于保证一个消费者能够“连续”取出10件产品，防止其他消费者在此期间进入。

------

### 2. 伪代码实现

```c
// 生产者进程
producer() {
    while (TRUE) {
        生产一件产品;
        
        P(empty);           // 1. 检查缓冲区是否有空位
        P(mutex);           // 2. 互斥地进入缓冲区
        
        将产品放入缓冲区;
        
        V(mutex);           // 3. 离开缓冲区
        V(full);            // 4. 通知消费者缓冲区多了产品
    }
}

// 消费者进程
consumer() {
    while (TRUE) {
        P(mutex_consumer);  // 【核心点】抢占连续取10件产品的权利
        
        for (int i = 0; i < 10; i++) {
            P(full);        // 5. 检查是否有产品（若无，则在此阻塞并握住mutex_consumer不放）
            P(mutex);       // 6. 互斥地进入缓冲区
            
            从缓冲区取出一件产品;
            
            V(mutex);       // 7. 离开缓冲区
            V(empty);       // 8. 释放一个空位信号
            
            消费产品;
        }
        
        V(mutex_consumer);  // 完成10件取用任务，释放权利给下一个消费者
    }
}
```

------

### 3. 关键逻辑详解

- 为什么要加 mutex_consumer？

  题目要求“连续取出 10 件”后其他消费者才能取。如果不加这个信号量，当第一个消费者取走第1件产品后，在它取第2件之前，CPU可能会切换到第二个消费者。加上 mutex_consumer 后，第一个消费者会“锁死”消费权限，直到它循环10次完成任务。

- 如果缓冲区里不到10件产品怎么办？

  如果此时 full 的值为5，第一个消费者进入循环，取走5件后，会在第6次的 P(full) 处阻塞。此时它依然握着 mutex_consumer。其他消费者无法取件，只有生产者可以继续放入产品。直到生产者补齐了剩下的5件，该消费者被唤醒并完成整个批次。这完美符合“连续取10件”的逻辑。

- P 操作顺序的重要性：

  在生产者中，必须先 P(empty) 再 P(mutex)；在消费者循环内，必须先 P(full) 再 P(mutex)。这是为了避免在缓冲区满或空时，进程带着 mutex 锁进入睡眠导致死锁。

**总结：** 这道题考察的是对“临界资源”范围的灵活定义。`mutex` 保护的是**单次读写动作**，而 `mutex_consumer` 保护的是**一整套读写序列**。

---

这是一个非常经典的困惑点。你可能会觉得应该有一个 `if (full >= 10)` 之类的判断，但**信号量（Semaphore）的精妙之处就在于：它的 P 操作本身就包含了“判断”和“等待”两个动作。**

在 2014 年这道考研真题的代码中，并没有“显式”地判断是否满 10 个，但通过 **`mutex_consumer`** 和 **`for` 循环** 的组合，从逻辑上完美实现了题目要求。

以下是深度解析为什么这样写就能生效，以及为什么不需要额外的“判断”：

### 1. 为什么 10 次 `P(full)` 就等于“判断是否满 10 个”？

信号量 `full` 代表缓冲区中的产品数量。

- 如果你执行一次 `P(full)`，你是在说：“我需要 1 个产品，如果没有，我就等。”
- 如果你在 `for` 循环里连续执行 10 次 `P(full)`，你实际上是在说：“**我总共需要 10 个产品。**”

关键点在于：

由于这 10 次 P(full) 是被包裹在 P(mutex_consumer) 里面的，这意味着在这个消费者“凑齐”10 个产品之前，它是绝对不会执行 V(mutex_consumer) 的。

### 2. 模拟“不满 10 个”的情况（看它如何不战而胜）

假设现在缓冲区里只有 **5 个** 产品，第一个消费者进程（$C_1$）进来了：

1. **抢锁**：$C_1$ 执行 `P(mutex_consumer)`，成功拿到消费权。
2. **取件循环**：
   - 前 5 次循环：`P(full)` 都能成功，$C_1$ 顺利取走 5 个产品。
   - **第 6 次循环**：$C_1$ 执行 `P(full)`。因为缓冲区空了（$full=0$），**$C_1$ 进程立即进入阻塞态（Sleep）**。
3. **锁的占有**：
   - **划重点**：$C_1$ 此时虽然在睡觉，但它**手里还死死攥着 `mutex_consumer` 这把锁**！
   - 其他消费者（$C_2, C_3$）如果想取件，会在 `P(mutex_consumer)` 处全部阻塞。
4. **生产者救场**：
   - 生产者进程可以继续往里放产品，每放一个就执行 `V(full)`。
   - 当生产者又放了 5 个产品后，$C_1$ 会被依次唤醒，完成剩下的 5 次取件循环。
5. **释放权利**：
   - 只有当 $C_1$ 完整地走完 10 次循环后，它才会执行 `V(mutex_consumer)`。
   - **结论**：此时，其他消费者才能进来。这就完美保证了“一个消费者连续取出 10 件产品后，其他消费者才可取”的题目要求。

### 3. 为什么不能写 `P(mutex_biggerthan10)`？

在操作系统的标准 PV 操作中，信号量只支持两种原子操作：

- **P(S)**：如果 $S > 0$，则 $S = S - 1$；否则阻塞。
- **V(S)**：$S = S + 1$；如果有等待者则唤醒。

**不存在**一种原语能直接判断“信号量是否大于某个特定值（如 10）”并原子性地扣减。如果你自己写 `if (full >= 10) { P(full)... }`，这在并发环境下是不安全的（在你判断完的一瞬间，产品可能被别人抢走了）。

### 4. 总结：代码设计的艺术

这道题的代码逻辑其实是：**利用“互斥锁”将多个“单次同步操作”打包成了一个“原子序列”**。

- **`P(full)`**：保证了“有东西才能取”的同步要求。
- **`for` 循环**：定义了“我要取多少”的数量要求。
- **`mutex_consumer`**：保证了这 10 次取件是“连续且不被干扰”的互斥要求。

所以，不需要写“判断是否满 10 个”，因为那 **10 次 `P(full)` 本身就是最严格的自动判断和自动等待机制**。
