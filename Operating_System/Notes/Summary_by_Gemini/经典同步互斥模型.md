### 1. 读者-写者问题 (Readers-Writers Problem)

核心场景：

一个共享文件（或数据库）。

- **读者 (Reader)**：只读不改，允许多个读者**同时**读。
- **写者 (Writer)**：既读也改，必须**独占**访问（写者在此时，不允许有其他读者或写者）。

难点：

如何实现“读-读允许，读-写互斥，写-写互斥”？我们需要一个计数器 count 来记录当前有几个读者。

- **第一个**进来的读者负责“锁门”（挡住写者）。
- **最后一个**离开的读者负责“开门”（唤醒写者）。

**经典代码（读者优先策略）：**

```c
semaphore rw = 1;      // 用于实现读写互斥、写写互斥
semaphore mutex = 1;   // 用于保护 count 变量的互斥修改
int count = 0;         // 记录当前正在读的读者数量

// 读者进程
reader() {
    while (TRUE) {
        P(mutex);           // 1. 申请修改 count 的权限
        if (count == 0) {
            P(rw);          // 2. 如果是第一个读者，负责把写者挡在门外
        }
        count++;
        V(mutex);           // 3. 释放 count 锁
        
        读文件...;
        
        P(mutex);           // 4. 申请修改 count 的权限
        count--;
        if (count == 0) {
            V(rw);          // 5. 如果是最后一个读者，负责给写者开门
        }
        V(mutex);
    }
}

// 写者进程
writer() {
    while (TRUE) {
        P(rw);              // 1. 申请读写锁（必须没人读、没人写才能进）
        
        写文件...;
        
        V(rw);              // 2. 释放读写锁
    }
}
```

### 2. 哲学家就餐问题 (Dining Philosophers Problem)

核心场景：

5 个哲学家围坐一张圆桌，每两个人之间有一根筷子（共 5 根）。哲学家只有同时拿到左右两根筷子才能吃饭。

难点：

死锁（Deadlock）。如果 5 个哲学家同时拿起左手的筷子，所有人都在等右手的筷子，结果谁也吃不上饭。

解决方案：

为了防止死锁，最简单的标准解法是：限制同时进餐（或试图进餐）的人数为 4 人。这样至少有 1 个人能拿到两根筷子。

**经典代码（限制人数法）：**

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1}; // 5根筷子
semaphore limit = 4;                      // 【关键】只允许4人同时抢筷子

// 第 i 号哲学家进程 (i = 0, 1, 2, 3, 4)
philosopher(int i) {
    while (TRUE) {
        思考...;
        
        P(limit);               // 1. 领取“准考证”，最多发4张，防止死锁
        
        P(chopstick[i]);        // 2. 拿左手筷子
        P(chopstick[(i+1)%5]);  // 3. 拿右手筷子（注意取模运算）
        
        进餐...;
        
        V(chopstick[i]);        // 4. 放回左手筷子
        V(chopstick[(i+1)%5]);  // 5. 放回右手筷子
        
        V(limit);               // 6. 交还“准考证”
    }
}
```

### 3. 理发师问题 (Sleeping Barber Problem)

核心场景：

理发店有 1 位理发师，1 把理发椅，N 把等候椅。

- 如果没有顾客，理发师睡觉（阻塞）。
- 顾客来了，如果理发师在睡觉，叫醒他。
- 如果理发师在忙，且有空椅子，顾客坐下等。
- **关键点**：如果椅子满了，顾客**直接离开**（这是与其他问题最大的不同，进程不会一直阻塞等待）。

难点：

需要准确判断“椅子满”的情况，并处理顾客直接退出的逻辑。

**经典代码：**

```c
semaphore customers = 0;  // 等待理发的顾客数（用来唤醒理发师）
semaphore barbers = 0;    // 理发师的状态（0=忙/睡，1=准备好剪发，用来阻塞顾客）
semaphore mutex = 1;      // 互斥访问 waiting 变量
int waiting = 0;          // 当前等待的人数
int CHAIRS = N;           // 椅子总数

// 理发师进程
barber() {
    while (TRUE) {
        P(customers);       // 1. 没顾客就睡觉（阻塞）
        
        P(mutex);           // 2. 互斥访问 waiting
        waiting--;          //    顾客少了一个（去剪发了）
        V(barbers);         // 3. 叫醒一个顾客过来剪发
        V(mutex);
        
        理发...;
    }
}

// 顾客进程
customer() {
    P(mutex);               // 1. 进店先看一眼椅子，要加锁
    
    if (waiting < CHAIRS) { // 2. 还有空位
        waiting++;          //    坐下等
        V(customers);       //    唤醒理发师（或者告诉理发师多了一个人）
        V(mutex);           //    释放锁，让后续逻辑继续
        
        P(barbers);         // 3. 等待理发师叫号（如果理发师在忙，就在这阻塞）
        接受理发...;
    } 
    else {
        V(mutex);           // 4. 没位子了，释放锁
        离开;               //    直接走人（不进行 P 操作）
    }
}
```

### 总结与记忆技巧

1. **读者-写者**：
   - **口诀**：第一个读者锁，最后一个读者开。
   - **特征**：`count` 变量必须用 `mutex` 保护。
2. **哲学家就餐**：
   - **口诀**：最多 4 人吃，或者 奇数拿左偶数拿右。
   - **特征**：必须打破“环路等待”条件以避免死锁。
3. **理发师**：
   - **口诀**：人满就走，人少就等。
   - **特征**：代码里一定有个 `if` 判断（资源满则退出），而不是无脑 `P`（阻塞）。
